## 1. 项目概述

### 1.1 项目背景
NAS 中存储约 10TB 电子元器件产品图片（约 2 万张），文件夹名称包含产品类别和 69 码后四位。现有查找方式低效：
- 在 ERP 中按 69 码后四位查找，再手动定位文件夹；
- 新人拿到无信息白底图时，几乎无法找到对应图片；
- 仅知道参数（如 “PCIE 4.0”）时，不能快速定位相关文件夹。

需要开发一个**本地离线桌面工具**，整合 ERP 搜索、文件夹浏览、以图搜图、OCR 关键字搜索，并预留 AI 助手扩展。

### 1.2 核心目标
- **统一搜索入口**：一个输入框支持文字、数字、拖图，自动分派搜索模式。
- **打通产品与图片**：导入 ERP Excel 后，自动通过 69 码后四位关联文件夹。
- **文件夹即产品单元**：搜索以文件夹为第一级结果，点击可查看其下所有图片。
- **极简离线优先**：核心检索功能完全离线运行；AI 助手为可选功能，需用户自行配置 API Key，无网络时提示"需要网络连接"。
- **预留 AI 助手**：通过快捷键截图 + OCR + DeepSeek API 分析屏幕内容并生成回复。

## 2. 功能需求

### 2.1 产品数据导入
- **触发**：菜单栏“导入产品”按钮。
- **输入**：Excel 文件（.xlsx/.xls），包含列：完整69码、产品名称、产品描述（列名可映射）。
- **行为**：解析并写入本地 SQLite 数据库，支持覆盖或追加。
- **输出**：产品表持久化，供所有搜索功能调用。

### 2.2 文件索引与 OCR 预处理
- **全量扫描**：
  - 用户指定 NAS 图片根目录（SMB 映射盘符或 UNC 路径）。
  - 遍历所有图片（.jpg/.jpeg/.png/.bmp/.webp）。
  - 记录文件 UNC 路径、所属文件夹名、修改时间。
  - 从文件夹名提取所有四位连续数字（`(?<!\d)\d{4}(?!\d)`，不用 `\b` 因为 Python 3 中 `\w` 包含中文导致边界失效）。
  - 与产品数据库匹配，建立文件夹-产品关联。
- **OCR 预处理**：
  - 对每张图片运行 PaddleOCR，提取图中文字（如 “PCIE 4.0”），存入数据库。
  - 首次全量后台运行（CPU 预估 10-20 小时，建议先取 100 张样本基准测试后校准；可通过多线程并行加速），增量时只处理新图片。
- **向量索引**：
  - 使用 CLIP (ViT-B/32) 提取每张图片 512 维特征，构建 FAISS HNSW 索引，支持增量添加。

### 2.3 统一搜索框
**位置**：窗口顶部，一个输入框。支持文字输入 + 拖入图片。

**触发时机（防抖）**：
- 用户停止输入 **400ms** 后自动触发搜索（无需按回车）。
- 每次击键重置计时器。按回车立即触发。
- 拖入图片 → 立即触发以图搜图。

**交互规则（三合一优先级）**：
- **仅有图** → 以图搜图模式。
- **仅有数字**（如 `1235`）→ 69 码渐进搜索模式。
- **仅有文字**（如 `USB`）→ 关键字搜索模式。
- **图 + 文字共存** → **交集搜索**：以图搜图结果 ∩ 文字搜索结果的文件夹交集，按 CLIP 相似度排序。
  - 交集为空时：展示以图搜图结果，标注"无匹配文字条件的文件夹"。

### 2.4 系统主题适配

- 启动时读取 Windows 注册表 `HKCU\Software\Microsoft\Windows\CurrentVersion\Themes\Personalize\AppsUseLightTheme` 判断系统暗色/亮色模式。
- 暗色模式 → Catppuccin Mocha 调色板；亮色模式 → Catppuccin Latte 调色板。
- 监听 `WM_SETTINGCHANGE` 消息（`lParam = "ImmersiveColorSet"`），用户切换系统主题时自动热切换。
- 所有颜色值集中在一个 `string.Template` 样式模板中，通过 `$token` 占位符按主题替换。
- 调试快捷键 `Ctrl+Shift+T` 强制切换主题（不改变系统设置，仅开发用）。

### 2.5 69码搜索（渐进匹配）

**不再仅匹配后四位**，而是随着用户输入的位数逐步缩小范围：

| 输入长度 | 匹配逻辑 | 示例 |
|---|---|---|
| 1-3 位 | `full_code LIKE '%输入'`（后缀模糊） | 输 "35" → 匹配 `...1235`, `...7835` 等 |
| 4 位 | `code_tail = 输入`（精确后四位，利用索引） | 输 "1235" → 精确匹配 |
| 5-12 位 | `full_code LIKE '%输入'`（后缀模糊，逐步逼近） | 输 "81235" → 匹配 `...781235` |
| 13 位 | `full_code = 输入`（完整69码精确匹配） | 完整码 → 唯一结果 |

**行为**：用户每多输入一位，"候选结果数"逐步收敛→逼近最终目标。输入到 4 位时给出精确候选，到 7-8 位时可能只剩 1-2 个结果。

### 2.6 关键字搜索（文件夹名 + OCR）

搜索结果始终 JOIN `product_folder_link` + `products`，统一三行格式。

**搜索优先级**：
1. `keyword_index` 表精确命中（扫描时预建的分词索引，毫秒级）
2. 文件夹名精确匹配 / 完全包含关键字
3. 文件夹名部分匹配
4. OCR 文本匹配（阶段 4 后生效）

### 2.7 结果展示

**所有搜索模式返回统一格式**：

有关联产品时显示三行，无关联时显示两行（仅文件夹名称 + 图片数量）：

| 行 | 内容 | 点击 📋 复制的内容 |
|---|---|---|
| 文件夹名称 | `文件夹名称：xxx` | 文件夹名原始文本（不含"文件夹名称："前缀） |
| 产品描述 | `产品描述：xxx`（有关联时）/ `图片数量：N`（无关联时） | 产品名原始文本 |
| 69码 | `69码：693456781235`（有关联时） | 完整 69 码数字 |

**以图搜图结果额外显示**：相似度百分比（如 `[相似度 94%]`）标在文件夹名后。

- 左侧：文件夹列表（按优先级排序），点击**整行文字区域**加载右侧缩略图。
- 右侧：单列缩略图网格，点击缩略图打开文件夹。
- 搜索框下方路径条显示当前文件夹完整路径（**点击路径条复制路径**）。
- 列表底部"📋 复制全部69码"按钮（**复制当前搜索结果中所有 69 码，换行分隔**）。
- 每行独立 📋 按钮：点击复制该行对应内容，不触发缩略图加载。

### 2.8 扫描时扩展建设

扫描遍历是成本最高的一趟操作。以下数据在扫描时”顺便”提取，几乎零额外 I/O 成本：

| 扩展项 | 说明 | 存储 | 实现阶段 |
|---|---|---|---|
| **文件夹名分词索引** | 扫描时对 `folder_name` 做中文/英文分词，写入 `keyword_index` 表 | `keyword_index(keyword, folder_path)` | 阶段 3 |
| **感知哈希 (pHash)** | 每张图计算 64 位 pHash，用于去重 + “找相似图” | `images.phash BIGINT` | 阶段 3 |
| **图片尺寸统计** | 记录每张图的宽、高、文件大小 | `images.width / height / file_size` | 阶段 3 |
| **错误日志** | 损坏/不支持的图片记录到 `data/scan_errors.log` | 文件日志 | 阶段 3 |
| **文件夹变化追踪** | 记录每次扫描的增/删/改 | `scan_log` 表 | 阶段 6 |

**辅助功能**：
- **增量扫描**：扫描新文件，追加索引。
- **重建全部索引**：全量重建所有数据和索引（保留产品数据）。
- **菜单栏按钮**：导入产品、增量扫描、重建索引、继续扫描（有待处理图片时显示）。

### 2.9 AI 助手模块（独立标签页）
- **形态**：标签页 `🤖 AI 助手`，与 `📦 图片检索` 并列。
- **功能**：
  - 全局快捷键触发截屏（全屏或区域），**快捷键由用户配置**（默认 `Ctrl+Shift+Z`）。
  - OCR 提取屏幕文字（支持多语言：马来语、泰语、英语、中文等）。
  - 用户输入指令（如”帮我回复”、”总结要点”），连同 OCR 文字发送至 DeepSeek API。
  - 返回结果自动复制到剪贴板，并展示在助手面板中。
- **隐私**：截图仅在本地 OCR 处理，仅文本片段上传 API。
- **网络依赖**：AI 助手需要 DeepSeek API 连接，为核心检索之外的可选功能。

## 3. 数据设计

### 3.1 SQLite 表结构

**产品表 `products`**
| 字段名      | 类型    | 约束                      | 说明                      |
|-------------|---------|---------------------------|---------------------------|
| id          | INTEGER | PRIMARY KEY               | 内部 ID                   |
| full_code   | TEXT    | NOT NULL UNIQUE           | 69码，**业务唯一键**       |
| code_tail   | TEXT    | NOT NULL, CHECK(length=4) | 后四位，由 INSERT 时 substr 自动填充，禁止手动修改 |
| name        | TEXT    | NOT NULL DEFAULT ''       | 产品名称                  |
| description | TEXT    | NOT NULL DEFAULT ''       | 产品描述                  |

> **code_tail 一致性规则**：所有 INSERT/UPDATE 操作必须通过 `import_products()` 方法，该方法内部自动计算 `code_tail = substr(full_code, -4)`。禁止直接 SQL 修改 `full_code` 或 `code_tail`。

**文件表 `images`**
| 字段名          | 类型    | 约束                    | 说明                               |
|-----------------|---------|-------------------------|------------------------------------|
| id              | INTEGER | PRIMARY KEY             | 内部 ID                            |
| full_unc        | TEXT    | NOT NULL UNIQUE         | 文件完整 UNC 路径                  |
| folder_path     | TEXT    | NOT NULL                | 所属文件夹路径                     |
| folder_name     | TEXT    | NOT NULL                | 文件夹名（派生冗余，加速查询）       |
| file_name       | TEXT    | NOT NULL                | 文件名                             |
| ocr_text        | TEXT    | NOT NULL DEFAULT ''     | OCR 文字                           |
| last_modified   | INTEGER | NOT NULL DEFAULT 0      | 文件修改时间戳                     |
| feature_extracted | INTEGER | NOT NULL DEFAULT 0 CHECK(feature_extracted IN (0,1)) | 特征提取标志 |
| thumbnail_path  | TEXT    | NOT NULL DEFAULT ''     | 缩略图路径                         |
| ocr_extracted   | INTEGER | NOT NULL DEFAULT 0 CHECK(ocr_extracted IN (0,1))    | OCR 标志       |
| phash           | INTEGER | NOT NULL DEFAULT 0      | 感知哈希（0=未计算）                |
| width           | INTEGER | NOT NULL DEFAULT 0      | 图片宽度                           |
| height          | INTEGER | NOT NULL DEFAULT 0      | 图片高度                           |
| file_size       | INTEGER | NOT NULL DEFAULT 0      | 文件大小 bytes                     |

**文件夹码表 `folder_codes`**
| 字段名      | 类型    | 约束                                         | 说明                     |
|-------------|---------|----------------------------------------------|--------------------------|
| id          | INTEGER | PRIMARY KEY                                  | 内部 ID                  |
| folder_path | TEXT    | NOT NULL                                     | 文件夹路径               |
| code_tail   | TEXT    | NOT NULL, CHECK(length(code_tail) BETWEEN 1 AND 4) | 提取的数字        |

| PRIMARY KEY | —       | (product_id, folder_path)          |                  |

**关键词索引表 `keyword_index`**
| 字段名      | 类型    | 约束                    | 说明                     |
|-------------|---------|-------------------------|--------------------------|
| keyword     | TEXT    | NOT NULL                | 分词后的关键词            |
| folder_path | TEXT    | NOT NULL                | 文件夹路径               |
| PRIMARY KEY | —       | (keyword, folder_path)  |                          |

**产品属性表 `product_attrs`（阶段 4 新增，EAV 模式）**
| 字段名      | 类型    | 约束                              | 说明                     |
|-------------|---------|-----------------------------------|--------------------------|
| id          | INTEGER | PRIMARY KEY                       |                          |
| image_id    | INTEGER | NOT NULL REFERENCES images(id)    | 关联图片                 |
| attr_name   | TEXT    | NOT NULL                          | 属性名（interface/material/form_factor） |

| 字段名      | 类型    | 约束                   | 说明                     |
|-------------|---------|------------------------|--------------------------|
| id          | INTEGER | PRIMARY KEY            |                          |
| scan_time   | REAL    | NOT NULL               | 扫描时间戳               |
| action      | TEXT    | NOT NULL CHECK(action IN ('added','deleted','modified')) | 变更类型 |
| file_path   | TEXT    | NOT NULL               | 文件路径                 |

**索引**：
- `images` 表：`folder_path`, `folder_name`, `full_unc`, `feature_extracted`, `ocr_extracted`
- `folder_codes` 表：`code_tail`, `folder_path`，`UNIQUE(folder_path, code_tail)`
- `products` 表：`full_code`（UNIQUE），`code_tail`
- `product_attrs` 表：`attr_name`, `image_id`
- `keyword_index` 表：`keyword`（PK 覆盖）

### 3.2 表设计规范（新增表必须遵守）

> 以 69 码为业务唯一键，以下规则确保后续阶段不加 NULL 判断、不做大规模迁移。

| # | 规则 | 说明 |
|---|---|---|
| 1 | **每一列必须有 DEFAULT** | 禁止 `TEXT` 无默认值。要么 `DEFAULT ''`，要么 `DEFAULT 0`。后期加列也必须有 DEFAULT |
| 2 | **flag 列加 CHECK** | `CHECK(col IN (0,1))`，防止脏数据 |
| 3 | **业务唯一键加 UNIQUE** | 不要依赖代码去重，表级别保证 |
| 4 | **派生列禁止手动写** | 如 `code_tail`，只由 `import_products()` 内部计算。文档注明"禁止直接 UPDATE" |
| 5 | **扩展属性走 EAV 关联表** | 不加列到实体表，用 `product_attrs(image_id, attr_name, attr_value)` 模式 |
| 6 | **外键在 SQLite 中用 REFERENCES 声明** | 虽然 SQLite 默认不强制 FK，但 `PRAGMA foreign_keys=ON` 后生效，且文档自解释 |
| 7 | **TEXT 主键 > 120 字符时考虑替代键** | `folder_path` 做 PK 会膨胀索引。优先用 INTEGER 代理键 + UNIQUE 约束 |
| 8 | **新增列必须同步更新 `init_db()` 迁移** | 用 `ALTER TABLE ADD COLUMN ... DEFAULT ...` + `try/except` 幂等 |

### 3.3 FAISS 向量索引文件
- 存储位置：`./data/faiss.index`
- 格式：`IndexHNSWFlat`，维度由当前模型决定（CLIP ViT-B/32 = 512）
- 向量 ID = `images.id - 1`（SQLite 从 1 开始，FAISS 从 0 开始）
- **索引元数据**：文件头存储 `model_name`（如 `"clip-ViT-B-32"`），加载时校验模型一致性

### 3.4 模型插件体系

> 核心原则：**模型是增强插件，不是核心功能。** 删除任意模型文件，app 仍正常启动。

所有模型代码放在 `models/` 目录：

```
models/
├── __init__.py              # 模型注册表 + 懒加载 + 自动发现
├── base.py                  # 抽象接口（不可删）
├── feature_extractors.py    # CLIP/DINOv2 等（可删）
├── query_analyzers.py       # LocalAnalyzer + CloudAnalyzer（可删）
└── ocr_engines.py           # PaddleOCR（可删，阶段4）
```

**注册表机制**：

```python
@register("feature_extractors")
class CLIPExtractor(BaseFeatureExtractor): ...

models.get_available("feature_extractors")  # → UI 下拉框数据源
models.get_instance("feature_extractors", "CLIPExtractor")  # → 懒加载，失败返回 None
```

- 文件删除 → 自动发现跳过 → 注册表为空 → UI 显示"未安装" → 核心功能不受影响
- 模型懒加载：只在第一次使用时实例化，失败不崩溃

**抽象接口**（`models/base.py`）：

| 接口 | 职责 | 阶段 |
|---|---|---|
| `BaseFeatureExtractor` | 图像 → 特征向量（CLIP/DINOv2） | 3 |
| `BaseQueryAnalyzer` | 自然语言 → `SearchIntent` 结构化参数 | 2.5 |
| `BaseOCREngine` | 图像 → 文字 | 4 |

### 3.5 QueryAnalyzer + SearchIntent

```python
@dataclass
class SearchIntent:
    mode: str            # "code" | "keyword" | "image"
    include_terms: list[str]
    exclude_terms: list[str]
    is_complex: bool     # True → 需要 API 语义解析

class BaseQueryAnalyzer(ABC):
    name: str
    @abstractmethod
    def analyze(self, query: str) -> SearchIntent: ...
```

**实现**：

| 实现类 | 文件 | 说明 |
|---|---|---|
| `LocalQueryAnalyzer` | `query_analyzers.py` | 无依赖，始终可用。数字→69码，文字→关键字 |
| `CloudQueryAnalyzer` | `query_analyzers.py` | 阶段 3.5 接入。复杂查询通过 API（DeepSeek/GPT/Claude）解析 |

**配置**：`config.json` 新增 `search.semantic_backend` + `search.semantic_endpoint`。

### 3.6 模型切换逻辑（特征提取）

> 核心原则：**向量和模型绑定。换模型 = 旧向量失效 = 提示重建。**

**启动时校验流程**：

```
加载 config.json → 读取 feature_model = "clip-ViT-B-32"
  ↓
检查 data/faiss.index 是否存在？
  ├─ 否 → 正常启动，等待用户扫描
  └─ 是 → 读取索引元数据中的 model_name
           ├─ 匹配 → 正常启动
           └─ 不匹配 → 弹出对话框：
                        "特征模型已变更 (旧: ViT-L-14 → 新: ViT-B-32)
                         旧特征向量与新模型不兼容。
                         [重建特征索引]  [稍后再说]"
                            ↓ 用户选"重建"
                         UPDATE images SET feature_extracted = 0
                         DELETE data/faiss.index
                         显示"继续扫描"按钮
```

**换模型影响范围**：

| 数据 | 换模型后 | 原因 |
|---|---|---|
| `images.feature_extracted` | **全部重置为 0** | 旧向量无效 |
| `data/faiss.index` | **删除重建** | 维度可能不同 |
| `images.thumbnail_path` | ✅ 保留 | 缩略图不依赖模型 |
| `images.ocr_text / ocr_extracted` | ✅ 保留 | OCR 不依赖模型 |
| `images.phash` | ✅ 保留 | pHash 不依赖模型 |
| `keyword_index` | ✅ 保留 | 分词不依赖模型 |
| `products / folder_codes / product_folder_link` | ✅ 保留 | 产品数据完全不相关 |

**中途切换保护**：
- 如果特征正在扫描中（`feature_extracted` 部分为 1、部分为 0），换模型时机已晚——所有 `feature_extracted=1` 的记录来自旧模型
- 换模型时统一重置：`UPDATE images SET feature_extracted = 0`
- 下次点"继续扫描"从断点开始，用新模型重新提取

**退回旧模型**：
- 如果旧 `faiss.index` 有备份（如 `faiss_clip-ViT-B-32.index.bak`），可恢复
- 没备份：`feature_extracted` 重置为 0，重新扫描

**设置对话框展显**：
- 下拉框选择模型（默认 CLIP ViT-B-32）
- 切换时弹警告："更换模型后需要重建特征索引。缩略图、OCR 等数据不受影响。"
- 保存后不立即重建，下次扫描时触发

### 3.7 配置文件
- `config.json`：保存 NAS 根路径、上次扫描时间、界面语言偏好、特征模型选择等。

### 阶段 3：以图搜图

**目标**：实现拖图搜索，打通 CLIP 特征提取与 FAISS 索引。支持断点续扫（关机/崩溃后可恢复）。不依赖 OCR，可独立交付。

**关键设计决策**：
- CLIP ViT-B/32 在 CPU 上约 0.3-0.5 秒/张，2 万张图片预估 2-3 小时。
- 必须支持"今天扫不完、明天接着扫"，否则长耗时任务无实用价值。
- 断点续扫通过 `images.feature_extracted` 标志位实现，每 100 张提交一次。
- **CLIP 白底图局限性**：CLIP 训练于自然场景图片，对"白底产品图 vs 场景图"的跨域匹配精度偏低。缓解方案：
  - **pHash 辅助**：扫描时计算每张图的感知哈希，搜索时用 Hamming 距离找相似/重复图（对同一产品不同角度/裁剪/压缩鲁棒）。
  - **OCR 兜底**：阶段 4 的 OCR 可提取产品图上的标签文字（如 69 码），文字匹配不受图像域影响，是白底图场景的最可靠兜底。
  - **多模型可选**：模型抽象层支持后续替换为工业场景更友好的模型（如 DINOv2）。

## 6. 版本控制与回退策略

### 6.1 Git 工作流
- **主分支**：`main`（保护分支，只接受合并请求，存储稳定版本）。
- **开发分支**：`dev`（日常开发）。
- **功能分支**：`feature/xxx`（每个阶段或功能在独立分支开发，完成后合并到 `dev`）。
- **标签**：每完成一个阶段里程碑，在 `dev` 合并到 `main` 后打 tag（如 `v0.1-data-scan`）。

### 6.2 回退方法
- 若某阶段新增功能出现严重问题，可基于上一阶段 tag 创建分支修复或继续开发，不影响已经稳定的版本。

## 8. 编码规范（所有阶段强制执行）

> 以下规则在每次提交代码前必须遵守，违反即视为 bug。

### 8.1 模型插件规范

- **位置**：所有模型实现必须放在 `models/` 目录，实现 `models/base.py` 中的抽象基类。
- **注册**：使用 `@register("category")` 装饰器注册，禁止在 UI 代码中硬编码模型类名。
- **懒加载**：通过 `models.get_instance(category, model_id)` 获取实例，失败返回 `None` 而非抛异常。
- **导入保护**：所有重型依赖（`torch`、`open_clip`、`paddleocr` 等）的 `import` 必须包裹在 `try/except ImportError` 中。
- **降级路径**：当模型不可用时，调用方必须提供 fallback（如 `LocalQueryAnalyzer` 兜底）。
- **删除容忍**：删除任意 `models/*.py`（除 `base.py` 和 `__init__.py`）后 app 必须能正常启动，受影响的仅是该模型对应的功能。
- **命名**：类名描述能力而非提供商（`CloudQueryAnalyzer` 而非 `DeepSeekQueryAnalyzer`）。

### 8.2 导入保护

- 任何可选模块的导入必须使用 `try/except ImportError`。
- 核心模块（`config`、`database`、`widgets`）不允许导入失败。
- `main.py` 入口必须在 3 秒内完成所有 import 并进入事件循环。

### 8.3 线程安全

- 扫描线程禁止直接操作 UI 组件，必须通过 `Signal` 跨线程通信。
- `Database` 实例跨线程共享时，每个方法调用 `_connect()` 打开独立连接。
- 禁止在扫描进行中启动第二个扫描（已实现守卫检查）。
- `QThread` 生命周期：`finished → quit → deleteLater`，禁止手动 `del` 或过早置 `None`。

### 8.4 信号/槽规范

- 连接信号时使用 `signal.connect(slot)`，禁止字符串式连接。
- 动态创建的信号连接必须在对象销毁前断开（或使用 `deleteLater` 自动管理）。
- 禁止对 QLabel 等控件 `monkey-patch mousePressEvent`，应使用 `eventFilter` 或子类化。

### 8.5 资源清理

- `QThread`：通过 `finished → deleteLater` 链路回收，禁止手动 `terminate()`。
- SQLite 连接：通过 `with self._connect()` 上下文管理器自动关闭。
- QPixmap 缓存：`_thumb_cache` 限制 3 个文件夹，超出时 LRU 淘汰。
- 文件句柄：所有 `open()` 使用 `with` 语句。

### 8.6 错误边界

- 所有用户触发的操作（导入、扫描、搜索）必须有 `try/except` 并弹出 `QMessageBox` 提示。
- 后台线程的异常通过 `error` 信号传递到主线程显示，禁止静默吞掉。
- 损坏/不支持的图片写入 `data/scan_errors.log`，不中断扫描。

### 8.7 异步与 GUI 线程安全（强制）

> 违反以下规则会导致 GUI 冻结、随机崩溃或未定义行为。

| # | 规则 | 说明 |
|---|---|---|
| 1 | **后台线程禁止创建 QWidget** | 包括 QMessageBox、QDialog。对话框必须通过信号触发，在主线程创建 |
| 2 | **所有 QObject 在 moveToThread 前创建，start() 前调用 moveToThread** | 确保 signals 以 Qt.QueuedConnection 跨线程调度，而非 DirectConnection |
| 3 | **>100ms 的操作必须在后台线程** | 图片解码、模型推理、文件 I/O 均适用。数据库查询当前均 <10ms，暂可保留主线程 |
| 4 | **网络请求用 QNetworkAccessManager** | 不另开线程，异步信号驱动。适用于 DeepSeek API 等 |
| 5 | **定时器回调捕获的 QWidget 必须用 QPointer 包装** | 防止列表清空后操作已删除的 C++ 对象 → `RuntimeError` |
| 6 | **WAL 模式保证读写并发** | SQLite PRAGMA journal_mode=WAL 已设置，写不阻塞读，读不阻塞写 |
| 7 | **config.save() 在后台线程调用需加锁** | 当前仅在 `_on_scan_finished`（后台线程）和设置对话框（主线程）调用，原子写入模式安全但不规范。后续加 `threading.Lock` |
| 8 | **模型加载在后台线程** | CLIP/PaddleOCR 加载权重需数秒，禁止阻塞主线程启动 |
| 9 | **ThreadGuard 模式** | 凡是"一个后台线程负责一种操作"的场景，用统一模式：`_cancel_xxx()` 在启动前调用 → quit + wait(500) → 然后创建新线程。禁止不 cancel 直接覆盖引用 |
| 10 | **回调加"请求指纹"校验** | 所有异步回调第一行必须校验"当前请求是否仍然有效"（如 `_thumb_folder` 标记、generation counter）。过期的回调直接 return |
