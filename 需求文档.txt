## 1. 项目概述

### 1.1 项目背景
NAS 中存储约 10TB 电子元器件产品图片（约 2 万张），文件夹名称包含产品类别和 69 码后四位。现有查找方式低效：
- 在 ERP 中按 69 码后四位查找，再手动定位文件夹；
- 新人拿到无信息白底图时，几乎无法找到对应图片；
- 仅知道参数（如 “PCIE 4.0”）时，不能快速定位相关文件夹。

需要开发一个**本地离线桌面工具**，整合 ERP 搜索、文件夹浏览、以图搜图、OCR 关键字搜索，并预留 AI 助手扩展。

### 1.2 核心目标
- **统一搜索入口**：一个输入框支持文字、数字、拖图，自动分派搜索模式。
- **打通产品与图片**：导入 ERP Excel 后，自动通过 69 码后四位关联文件夹。
- **文件夹即产品单元**：搜索以文件夹为第一级结果，点击可查看其下所有图片。
- **极简离线优先**：核心检索功能完全离线运行；AI 助手为可选功能，需用户自行配置 API Key，无网络时提示"需要网络连接"。
- **预留 AI 助手**：通过快捷键截图 + OCR + DeepSeek API 分析屏幕内容并生成回复。

## 2. 功能需求

### 2.1 产品数据导入
- **触发**：菜单栏“导入产品”按钮。
- **输入**：Excel 文件（.xlsx/.xls），包含列：完整69码、产品名称、产品描述（列名可映射）。
- **行为**：解析并写入本地 SQLite 数据库，支持覆盖或追加。
- **输出**：产品表持久化，供所有搜索功能调用。

### 2.2 文件索引与 OCR 预处理
- **全量扫描**：
  - 用户指定 NAS 图片根目录（SMB 映射盘符或 UNC 路径）。
  - 遍历所有图片（.jpg/.jpeg/.png/.bmp/.webp）。
  - 记录文件 UNC 路径、所属文件夹名、修改时间。
  - 从文件夹名提取所有四位连续数字（`\b\d{4}\b`）作为 69 码后四位。
  - 与产品数据库匹配，建立文件夹-产品关联。
- **OCR 预处理**：
  - 对每张图片运行 PaddleOCR，提取图中文字（如 “PCIE 4.0”），存入数据库。
  - 首次全量后台运行（CPU 预估 10-20 小时，建议先取 100 张样本基准测试后校准；可通过多线程并行加速），增量时只处理新图片。
- **向量索引**：
  - 使用 CLIP (ViT-B/32) 提取每张图片 512 维特征，构建 FAISS HNSW 索引，支持增量添加。

### 2.3 统一搜索框
**位置**：窗口顶部，一个输入框。支持文字输入 + 拖入图片。

**触发时机（防抖）**：
- 用户停止输入 **400ms** 后自动触发搜索（无需按回车）。
- 每次击键重置计时器。按回车立即触发。
- 拖入图片 → 立即触发以图搜图。

**交互规则（三合一优先级）**：
- **仅有图** → 以图搜图模式。
- **仅有数字**（如 `1235`）→ 69 码渐进搜索模式。
- **仅有文字**（如 `USB`）→ 关键字搜索模式。
- **图 + 文字共存** → **交集搜索**：以图搜图结果 ∩ 文字搜索结果的文件夹交集，按 CLIP 相似度排序。
  - 交集为空时：展示以图搜图结果，标注"无匹配文字条件的文件夹"。

### 2.5 69码搜索（渐进匹配）

**不再仅匹配后四位**，而是随着用户输入的位数逐步缩小范围：

| 输入长度 | 匹配逻辑 | 示例 |
|---|---|---|
| 1-3 位 | `full_code LIKE '%输入'`（后缀模糊） | 输 "35" → 匹配 `...1235`, `...7835` 等 |
| 4 位 | `code_tail = 输入`（精确后四位，利用索引） | 输 "1235" → 精确匹配 |
| 5-12 位 | `full_code LIKE '%输入'`（后缀模糊，逐步逼近） | 输 "81235" → 匹配 `...781235` |
| 13 位 | `full_code = 输入`（完整69码精确匹配） | 完整码 → 唯一结果 |

**行为**：用户每多输入一位，"候选结果数"逐步收敛→逼近最终目标。输入到 4 位时给出精确候选，到 7-8 位时可能只剩 1-2 个结果。

### 2.6 关键字搜索（文件夹名 + OCR）

搜索结果始终 JOIN `product_folder_link` + `products`，统一三行格式。

**搜索优先级**：
1. `keyword_index` 表精确命中（扫描时预建的分词索引，毫秒级）
2. 文件夹名精确匹配 / 完全包含关键字
3. 文件夹名部分匹配
4. OCR 文本匹配（阶段 4 后生效）

### 2.7 结果展示

**所有搜索模式返回统一格式**，左侧列表每项至少两行、最多三行：

| 行 | 内容 | 可复制 |
|---|---|---|
| 文件夹名称 | `文件夹名称：xxx` | ✅ |
| 产品描述 | `产品描述：xxx`（有关联时）/ `图片数量：N`（无关联时） | ✅（有关联时） |
| 69码 | `69码：693456781235`（有关联时） | ✅ |

**以图搜图结果额外显示**：相似度百分比（如 `[相似度 94%]`）标在文件夹名后。

- 左侧：文件夹列表（按优先级排序），点击行加载右侧缩略图。
- 右侧：单列缩略图网格，点击打开文件夹。
- 搜索框下方路径条显示当前文件夹完整路径（点击复制）。
- 列表底部"📋 复制全部69码"按钮。

### 2.8 扫描时扩展建设

扫描遍历是成本最高的一趟操作。以下数据在扫描时”顺便”提取，几乎零额外 I/O 成本：

| 扩展项 | 说明 | 存储 | 实现阶段 |
|---|---|---|---|
| **文件夹名分词索引** | 扫描时对 `folder_name` 做中文/英文分词，写入 `keyword_index` 表 | `keyword_index(keyword, folder_path)` | 阶段 3 |
| **感知哈希 (pHash)** | 每张图计算 64 位 pHash，用于去重 + “找相似图” | `images.phash BIGINT` | 阶段 3 |
| **图片尺寸统计** | 记录每张图的宽、高、文件大小 | `images.width / height / file_size` | 阶段 3 |
| **错误日志** | 损坏/不支持的图片记录到 `data/scan_errors.log` | 文件日志 | 阶段 3 |
| **文件夹变化追踪** | 记录每次扫描的增/删/改 | `scan_log` 表 | 阶段 6 |

**辅助功能**：
- **增量扫描**：扫描新文件，追加索引。
- **重建全部索引**：全量重建所有数据和索引（保留产品数据）。
- **菜单栏按钮**：导入产品、增量扫描、重建索引、继续扫描（有待处理图片时显示）。

### 2.9 AI 助手模块（独立标签页）
- **形态**：标签页 `🤖 AI 助手`，与 `📦 图片检索` 并列。
- **功能**：
  - 全局快捷键触发截屏（全屏或区域），**快捷键由用户配置**（默认 `Ctrl+Shift+Z`）。
  - OCR 提取屏幕文字（支持多语言：马来语、泰语、英语、中文等）。
  - 用户输入指令（如”帮我回复”、”总结要点”），连同 OCR 文字发送至 DeepSeek API。
  - 返回结果自动复制到剪贴板，并展示在助手面板中。
- **隐私**：截图仅在本地 OCR 处理，仅文本片段上传 API。
- **网络依赖**：AI 助手需要 DeepSeek API 连接，为核心检索之外的可选功能。

## 3. 数据设计

### 3.1 SQLite 表结构

**产品表 `products`**
| 字段名      | 类型    | 说明                      |
|-------------|---------|---------------------------|
| id          | INTEGER | 主键                      |
| full_code   | TEXT    | 完整69码，唯一索引         |
| code_tail   | TEXT    | 后四位，有索引             |
| name        | TEXT    | 产品名称                  |
| description | TEXT    | 产品描述                  |

**文件表 `images`**
| 字段名          | 类型    | 说明                               |
|-----------------|---------|------------------------------------|
| id              | INTEGER | 主键，对应 FAISS 向量 ID（0起始）  |
| full_unc        | TEXT    | 文件完整 UNC 路径，唯一            |
| folder_path     | TEXT    | 所属文件夹路径（从根目录起）        |
| folder_name     | TEXT    | 最末级文件夹名                     |
| file_name       | TEXT    | 文件名（不含路径）                 |
| ocr_text        | TEXT    | OCR 提取的文字，可空               |
| last_modified   | INTEGER | 文件修改时间戳                     |
| feature_extracted | INTEGER | 0=未提取特征, 1=已完成（断点续扫） |
| thumbnail_path  | TEXT    | 缩略图缓存路径（空=未生成）         |
| ocr_extracted   | INTEGER | 0=未OCR, 1=已完成（阶段4）          |
| phash           | INTEGER | 64位感知哈希（阶段3）               |
| width           | INTEGER | 图片宽度 px（阶段3）                |
| height          | INTEGER | 图片高度 px（阶段3）                |
| file_size       | INTEGER | 文件大小 bytes（阶段3）             |

**文件夹码表 `folder_codes`**
| 字段名      | 类型    | 说明                     |
|-------------|---------|--------------------------|
| id          | INTEGER | 主键                     |
| folder_path | TEXT    | 文件夹路径               |
| code_tail   | TEXT    | 提取的四位数字（如 1235）|

**关键词索引表 `keyword_index`（阶段 3 新增）**
| 字段名      | 类型    | 说明                     |
|-------------|---------|--------------------------|
| keyword     | TEXT    | 分词后的关键词            |
| folder_path | TEXT    | 包含该关键词的文件夹路径   |
| PRIMARY KEY | (keyword, folder_path) |           |

**扫描日志表 `scan_log`（阶段 6 新增）**
| 字段名      | 类型    | 说明                     |
|-------------|---------|--------------------------|
| id          | INTEGER | 主键                     |
| scan_time   | REAL    | 扫描时间戳               |
| action      | TEXT    | added / deleted / modified |
| file_path   | TEXT    | 文件路径                 |
| folder_path | TEXT    | 所属文件夹               |

注：一个文件夹可能提取出多个四位数字，一个四位数字可能匹配多个产品。去歧义逻辑在查询时通过文件夹名与产品名的文本相似度加权实现，不在此表中处理。

**产品-文件夹关联表 `product_folder_link`（可选缓存）**
| 字段名      | 类型    | 说明             |
|-------------|---------|------------------|
| product_id  | INTEGER | 关联 products.id |
| folder_path | TEXT    | 文件夹路径       |

**索引**：
- `images` 表：`folder_path`, `folder_name`, `full_unc` 建立索引。
- `folder_codes` 表：`code_tail`, `folder_path` 索引。
- `products` 表：`full_code` 索引（自动通过唯一约束）。

### 3.2 FAISS 向量索引文件
- 存储位置：`./data/faiss.index`
- 格式：`IndexHNSWFlat`，维度由当前模型决定（CLIP ViT-B/32 = 512）
- 向量 ID 与 `images.id` 严格一致
- **索引元数据**：文件头存储 `model_name`（如 `"clip-ViT-B-32"`），加载时校验模型一致性

### 3.3 特征提取模型抽象层

> 设计目标：允许用户随时更换特征提取模型，不影响其他数据（缩略图、OCR、产品数据）。

**`BaseFeatureExtractor` 抽象基类**：

```python
class BaseFeatureExtractor(ABC):
    @property
    def dim(self) -> int: ...           # 输出向量维度
    @property
    def model_name(self) -> str: ...    # 唯一标识，写入 FAISS 元数据

    @abstractmethod
    def extract_batch(self, paths: list[str]) -> np.ndarray: ...
    @abstractmethod
    def extract_single(self, path: str) -> np.ndarray: ...
```

**内置实现**：

| 实现类 | 维度 | 模型名 | 说明 |
|---|---|---|---|
| `CLIPExtractor` | 512 | `"clip-ViT-B-32"` | 默认，CPU 平衡 |
| `CLIPLargeExtractor` | 768 | `"clip-ViT-L-14"` | 精度更高，慢 3x |

**配置**：`config.json` 新增字段 `feature_model: "clip-ViT-B-32"`。

### 3.4 模型切换逻辑

> 核心原则：**向量和模型绑定。换模型 = 旧向量失效 = 提示重建。**

**启动时校验流程**：

```
加载 config.json → 读取 feature_model = "clip-ViT-B-32"
  ↓
检查 data/faiss.index 是否存在？
  ├─ 否 → 正常启动，等待用户扫描
  └─ 是 → 读取索引元数据中的 model_name
           ├─ 匹配 → 正常启动
           └─ 不匹配 → 弹出对话框：
                        "特征模型已变更 (旧: ViT-L-14 → 新: ViT-B-32)
                         旧特征向量与新模型不兼容。
                         [重建特征索引]  [稍后再说]"
                            ↓ 用户选"重建"
                         UPDATE images SET feature_extracted = 0
                         DELETE data/faiss.index
                         显示"继续扫描"按钮
```

**换模型影响范围**：

| 数据 | 换模型后 | 原因 |
|---|---|---|
| `images.feature_extracted` | **全部重置为 0** | 旧向量无效 |
| `data/faiss.index` | **删除重建** | 维度可能不同 |
| `images.thumbnail_path` | ✅ 保留 | 缩略图不依赖模型 |
| `images.ocr_text / ocr_extracted` | ✅ 保留 | OCR 不依赖模型 |
| `images.phash` | ✅ 保留 | pHash 不依赖模型 |
| `keyword_index` | ✅ 保留 | 分词不依赖模型 |
| `products / folder_codes / product_folder_link` | ✅ 保留 | 产品数据完全不相关 |

**中途切换保护**：
- 如果特征正在扫描中（`feature_extracted` 部分为 1、部分为 0），换模型时机已晚——所有 `feature_extracted=1` 的记录来自旧模型
- 换模型时统一重置：`UPDATE images SET feature_extracted = 0`
- 下次点"继续扫描"从断点开始，用新模型重新提取

**退回旧模型**：
- 如果旧 `faiss.index` 有备份（如 `faiss_clip-ViT-B-32.index.bak`），可恢复
- 没备份：`feature_extracted` 重置为 0，重新扫描

**设置对话框展显**：
- 下拉框选择模型（默认 CLIP ViT-B-32）
- 切换时弹警告："更换模型后需要重建特征索引。缩略图、OCR 等数据不受影响。"
- 保存后不立即重建，下次扫描时触发

### 3.3 配置文件
- `config.yaml`：保存 NAS 根路径、上次扫描时间、界面语言偏好等。

## 4. 技术选型

| 模块           | 技术                              | 说明                                     |
|----------------|-----------------------------------|------------------------------------------|
| GUI            | PySide6                           | 悬浮窗口、标签页、系统托盘、全局热键、拖拽支持 |

| 打包           | PyInstaller (--onedir)            | 便携式文件夹分发，zip 压缩包解压即用，避免单文件 exe 体积过大（2-4GB）和启动慢（30-60s 解压）的问题 |
| 版本控制       | Git                               | 分阶段打 tag，方便回退                    |

- **里程碑 tag**: `v0.2-text-search`

### 阶段 3：以图搜图

**目标**：实现拖图搜索，打通 CLIP 特征提取与 FAISS 索引。支持断点续扫（关机/崩溃后可恢复）。不依赖 OCR，可独立交付。

**关键设计决策**：
- CLIP ViT-B/32 在 CPU 上约 0.3-0.5 秒/张，2 万张图片预估 2-3 小时。
- 必须支持"今天扫不完、明天接着扫"，否则长耗时任务无实用价值。
- 断点续扫通过 `images.feature_extracted` 标志位实现，每 100 张提交一次。

#### 3.1 数据库变更

- [ ] `images` 表新增列：`feature_extracted`, `thumbnail_path`, `phash`, `width`, `height`, `file_size`
- [ ] 创建索引 `idx_images_feature_extracted ON images(feature_extracted)`
- [ ] 新建 `keyword_index` 表（文件夹名分词倒排索引）
- [ ] 扫描错误日志 `data/scan_errors.log`

#### 3.2 依赖安装

#### 3.3 FeatureExtractor（抽象层）

- [ ] 定义 `BaseFeatureExtractor` 抽象基类（`feature_extractor.py`）：
  ```python
  class BaseFeatureExtractor(ABC):
      dim: int           # 输出向量维度
      model_name: str    # 唯一标识（存入 FAISS 元数据 + config）

      @abstractmethod
      def extract_batch(self, paths: list[str]) -> np.ndarray: ...
      @abstractmethod
      def extract_single(self, path: str) -> np.ndarray: ...
  ```
- [ ] 实现 `CLIPExtractor(ViT-B-32)`：默认实现，dim=512
- [ ] 支持 CPU / CUDA 切换（`config.json` 的 `device` 字段）

#### 3.4 断点续扫（核心）

- [ ] 扫描逻辑改为**三阶段流水线**：
  ```
  Phase A: 文件发现（复用现有 os.walk）
  Phase B: 特征提取 + FAISS 写入（新）
  Phase C: 缩略图生成（并行）
  ```

- [ ] 断点续扫实现：
  ```
  for each image in images WHERE feature_extracted = 0:
      1. 加载图片 → CLIP 推理 → 得到 512d 向量
      2. faiss_index.add(vector)
      3. 生成缩略图 (200px) → 存到 thumbnails/{md5}.jpg
      4. UPDATE images SET feature_extracted=1, thumbnail_path=...
      5. 每 100 张: conn.commit() + faiss.write_index("data/faiss.index")
  ```

- [ ] 用户可随时关闭窗口 → 下次启动点击"继续扫描"从断点恢复
- [ ] "继续扫描"按钮在 `feature_extracted = 0` 的图片数量 > 0 时显示
- [ ] 进度显示："已完成 3500 / 20000 (17%)，预计剩余 2h15m"（基于前 100 张耗时推算）

#### 3.5 FAISS 索引管理

- [ ] 索引类型：`IndexHNSWFlat`（维度 512，HNSW 图 M=32）
- [ ] 存储位置：`data/faiss.index`
- [ ] 向量 ID 与 `images.id` 严格对应（id=0 为 FAISS 向量 0）
- [ ] 加载逻辑：
  ```
  if exists("data/faiss.index"):
      index = faiss.read_index("data/faiss.index")
  else:
      index = faiss.IndexHNSWFlat(512, 32)
  ```
- [ ] 每 100 条写入一次 `faiss.write_index()`，防止崩溃丢失全部向量
- [ ] 全量重建索引时删除 `data/faiss.index`，清空 `images.feature_extracted` 重置为 0

## 6. 版本控制与回退策略

### 6.1 Git 工作流
- **主分支**：`main`（保护分支，只接受合并请求，存储稳定版本）。
- **开发分支**：`dev`（日常开发）。
- **功能分支**：`feature/xxx`（每个阶段或功能在独立分支开发，完成后合并到 `dev`）。
- **标签**：每完成一个阶段里程碑，在 `dev` 合并到 `main` 后打 tag（如 `v0.1-data-scan`）。

### 6.2 回退方法
- 若某阶段新增功能出现严重问题，可基于上一阶段 tag 创建分支修复或继续开发，不影响已经稳定的版本。
- 示例：
  ```bash
  git checkout v0.2-text-search -b hotfix/xxx   # 回到阶段2（文本搜索）状态
  git checkout v0.3-image-search -b hotfix/xxx  # 回到阶段3（以图搜图）状态
  ```

### 6.3 数据兼容性
- 每个阶段的数据库结构可能微调，需在代码中编写**数据库迁移脚本**（简单版本号检查，执行 `ALTER TABLE` 等）。
- 配置文件也可通过版本号进行兼容处理。

### 7.1 搜索结果显示格式化（main.py `SearchTab`）

列表项由两个独立字段组成，分别通过静态方法格式化：

| 方法 | 职责 | 扩展方向 |
|------|------|----------|
| `_fmt_folder_label(folder_name)` | 文件夹名显示文本 | AI 简化文件夹名、翻译、提取关键信息 |
| `_fmt_product_label(product_name, full_code)` | 产品标签显示文本 | AI 摘要产品描述、翻译名称 |

每个 `QListWidgetItem` 存储结构化数据：
- `UserRole`：`folder_path`（供缩略图加载）
- `UserRole+1`：`full_code` 69码（供复制操作）

后续阶段只需修改上述两个静态方法的返回值，无需改动 UI 框架。
