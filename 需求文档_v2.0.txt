## 1. 项目概述

### 1.1 项目背景
NAS 中存储大量电子元器件产品图片，主要目录为：
  - `K:\1.图片\2.转接卡`（约 642 个文件夹）
  - `K:\1.图片\13.PCIE延长线`（约 178 个文件夹）

CSV 商品表（店小秘导出）共 **1405 个产品**，其中：
  - **1225 个**使用 69 码作为主标识（87%）
  - **180 个**使用自定义 SKU（13%）
  - 产品分类覆盖 20+ 个类别：转接卡、延长线、硬盘盒、散热片、数据线、风扇、支架、水晶头、模块、护套、快充、电源、转接板、网线钳、阵列卡、增高卡、拆分卡 等

实际目录结构远比最初设想的复杂：
  - 一个文件夹常包含多个 SKU（多颜色、多尺寸、多版本）
  - 一个产品文件夹下有多个语言子文件夹（中文/英语/西班牙语/俄语/韩语/印尼语/泰语/法文）
  - 存在渲染图、实拍图、亚马逊素材、停售素材等用途子文件夹
  - 文件夹名中的 69 码后四位可能缺失、多个、或位于子文件夹名中

现有查找方式低效：
  - 在 ERP 中按 69 码后四位查找，再手动定位文件夹；
  - 新人拿到无信息白底图时，几乎无法找到对应图片；
  - 仅知道参数（如 "PCIE 4.0"）时，不能快速定位相关文件夹。

### 1.2 核心目标
- **统一搜索入口**：一个输入框支持文字、数字、拖图，自动分派搜索模式。
- **产品组为核心**：一个文件夹 = 一个产品组（product_group），组下可含多个 SKU 变体。
- **图片集管理**：同一产品组下的多语言/多用途图片按 image_set 组织，选择性索引。
- **打通产品与图片**：导入 CSV 后，通过 69 码后四位 + 文件夹名分词关联产品组。
- **多数据源融合**：CSV（主数据）+ LLM JSONL（图片分析）+ 文件夹结构 → SQLite。
- **离线优先**：核心检索完全离线运行；AI 助手为可选功能，需用户自行配置 API Key。
- **LLM 前置识别**：AI 分析输出 products.jsonl，扫描器离线运行，两者解耦。

### 1.3 架构概览

```
  [CSV 商品表] ──┐
                 ├──→ [产品库构建] ──→ [SQLite 数据库]
  [LLM JSONL] ──┘                            ↑
                                              │
  [NAS 扫描] ──→ [image_sets / images] ──────┘
                 [PaddleOCR 文字] ──→ [keyword_index]
                 [CLIP 特征] ──→ [FAISS 索引]
                      │
                      ↓
              [SearchDispatcher]
                      │
        ┌───────┬───────┼───────┐
        ↓       ↓       ↓       ↓
      69码    关键字   以图搜图  交集搜索
                      │
                      ↓
         ┌─────────────────────────┐
         │    搜索结果（产品优先）    │
         │  ┌──────────┬─────────┐ │
         │  │ 左侧      │ 右侧     │ │
         │  │ 产品卡片  │ 详情面板  │ │
         │  │ ·主图     │ ·全部图片 │ │
         │  │ ·69码    │ ·规格参数 │ │

         │  └──────────┴─────────┘ │
         └─────────────────────────┘
```

## 2. 真实数据特征

### 2.1 关键发现（基于 mock-data 验证）

**(1) 文件夹层级深**
产品根文件夹下通常包含：
  - 中文 / 英语 / 西班牙语 / 俄语 / 韩语 / 印尼语 / 泰语 / 法文（语言子文件夹）
  - 渲染图
  - 实拍图
  - 亚马逊
  - 停售 / 停用
  - 主图 / 详情

**(2) 多 SKU 合并是常态**
典型文件夹名：
  - `10.型号：冷雨燕 黑 型号：冷雨燕 红 2180 1961`
  - `15.型号：保护板一号 型号：保护板二号 6638 6645`
  - `39.增高卡 型号：ZX51827 型号：ZX51818 型号：ZX51890 3628 3321 3611`

**(3) 多语言图片本质重复**
英语/1.jpg、西班牙语/1.jpg、俄语/1.jpg 往往是同一张图换文字，不应全部做 CLIP 向量，否则浪费空间和算力。

**(4) 69 码来源多样**
- 文件夹名中的后四位
- 子文件夹名中的后四位
- CSV 中的完整 69 码
- 图片上的条码（由 LLM 识别）

### 2.2 方案调整清单

| 原方案假设               | 实际情况                     | 调整                          |
|--------------------------|------------------------------|-------------------------------|
| 文件夹 = 产品            | 文件夹 = 产品组（多 SKU）    | 新增 product_groups 表        |
| 文件夹下图片直接属产品   | 分语言/渲染/实拍等子文件夹   | 新增 image_sets 表            |
| LLM JSONL 一行一个产品   | 需要表达一组变体             | JSONL 支持 products 数组      |
| 只做 CLIP 和文件名索引   | 多语言重复图浪费空间         | 按 image_set 类型选择性索引   |
| 69码从文件夹名提取       | 多个后四位或没有             | 结合 CSV 数据反向匹配         |
| Excel 只导入 4 个字段    | CSV 有长宽高/重量/海关编码   | 充分利用 CSV 字段填充 specs   |
| 搜索返回产品列表         | 变体过多会刷屏               | 默认按产品组聚合，可展开变体  |

## 3. 功能需求

### 3.1 产品数据导入

支持三种数据来源：

**(1) CSV 商品表导入**
- 读取 `.csv` 文件，列名可映射
- 关键字段：`sku(必填)`、`平台SKU`、`中文名称`、`英文名称`、`商品净重(g)`、`长(cm)`、`宽(cm)`、`高(cm)`、`英文报关`、`中文报关`、`海关编码`、`开发员`
- 从名称自动提取：颜色、接口类型（X1/X4/X16/M.2/NVMe）、尺寸长度（2280/2242/50CM/20CM）、PCIe 版本（4.0/5.0/3.0）
- 长/宽/高/重量/海关编码直接写入 `product_specs`
- 自动识别产品类别（不硬编码关键词，基于名称模式匹配）

**(2) LLM 产品库导入**
- 扫描器启动时自动检测根目录下的 `products.jsonl`
- 读取并合并到 SQLite
- 支持增量更新（通过文件修改时间判断）

**(3) 文件夹结构导入**
- 扫描 NAS 目录，自动识别产品根文件夹、image_set 子文件夹
- 未匹配到 CSV/JSONL 的文件夹标记为"未关联产品组"，仍可被搜索

### 3.2 文件索引（离线扫描）

扫描器只做轻量操作：
  - 识别产品根文件夹
  - 识别 image_set 子文件夹（语言/渲染/实拍/亚马逊/停售等）
  - 记录文件 UNC 路径、所属图片集、修改时间
  - 从文件夹名提取所有 4 位连续数字，作为候选 69 码后四位
  - 对主语言集/实拍图做 CLIP 特征提取
  - 生成缩略图（200px）
  - 计算 pHash、记录宽高和文件大小
  - 对产品根文件夹名做分词索引

### 3.3 图片集类型识别

扫描器自动识别以下 folder_name 模式：

| 文件夹名                  | set_type      | language | 扫描优先级 | 索引策略   |
|---------------------------|---------------|----------|-----------|------------|
| 中文                      | language      | zh       | 1         | 优先索引   |
| 英语 / 英文               | language      | en       | 1         | 优先索引   |
| 西班牙语                  | language      | es       | 1         | 优先索引   |
| 俄语                      | language      | ru       | 2         | 备选索引   |
| 韩语                      | language      | ko       | 2         | 备选索引   |
| 印尼语                    | language      | id       | 2         | 备选索引   |
| 泰语                      | language      | th       | 2         | 备选索引   |
| 法文                      | language      | fr       | 2         | 备选索引   |
| 渲染图                    | render        | -        | 3         | 可选索引   |
| 实拍图                    | real          | -        | 2         | 建议索引   |
| 主图                      | primary       | -        | 1         | 优先索引   |
| 详情                      | detail        | -        | 2         | 建议索引   |
| 亚马逊                    | amazon        | -        | 3         | 可选索引   |
| 停售 / 停用 / 暂未售      | discontinued  | -        | 0         | 默认不索引 |

### 3.4 统一搜索框

- 顶部单一输入框，支持文字输入 + 拖入图片
- 400ms 防抖自动触发，回车立即触发
- 拖入图片立即触发以图搜图

搜索模式分发：
  - 仅有图        → 以图搜图
  - 仅有数字      → 69 码渐进搜索
  - 仅有文字      → 关键字搜索
  - 图 + 文字     → 交集搜索

### 3.5 69 码渐进搜索

| 输入长度 | 匹配逻辑                              |
|----------|---------------------------------------|
| 1-3 位   | `full_code LIKE '%输入'`（后缀模糊）    |
| 4 位     | `code_tail = 输入`（精确后四位）        |
| 5-12 位  | `full_code LIKE '%输入'`（逐步逼近）    |
| 13 位    | `full_code = 输入`（完整码精确）        |

### 3.6 关键字搜索（优先级）

1. `keyword_index` 表精确命中（文件夹名分词，毫秒级）
2. `product_keywords` 表命中（产品名/描述/规格/标签）
3. `product_specs` 规格值匹配
4. OCR 文本匹配（`images.ocr_text LIKE '%keyword%'`）
5. 产品名部分匹配
6. 文件夹名部分匹配

#### 3.7.2 搜索栏（SearchBar）

**文件**：`search_tab.py` → `SearchBar` 类

单个 `QLineEdit` + 模式标签 + 清除按钮。

| 输入类型 | 检测方式 | 模式标签 | 派发目标 |
|----------|----------|----------|----------|
| 纯数字（如 `2233`） | `text.isdigit()` | `[69码]` 蓝色 | `database.search_products()` 渐进匹配 |
| 含文字（如 `蜻蜓`） | 非纯数字 | `[关键词]` 绿色 | `database.search_products()` 关键字搜索 |
| 拖入图片 | 后续实现 | `[以图搜图]` 紫色 | CLIP + FAISS |
| 图片 + 文字 | 后续实现 | `[交集]` 橙色 | FAISS ∩ 文字结果 |

**防抖**：400ms（`QTimer.singleShot`），回车立即触发。

**为什么不做分类/排序下拉**：卖家最常用的操作是输入 69 码直接定位产品。分类筛选会增加操作步骤，且当前产品数（1405）不足以需要分类过滤。搜索结果的匹配度排序已经足够。

#### 3.7.3 左侧：产品卡片列表（ProductList）

**文件**：`search_tab.py` → `ProductCard` 类 + `ProductList` 类

每个搜索结果 = **一个产品 SKU**，不是文件夹、不是产品组。

**卡片结构**：
```
┌──────────────────────────────────┐
│ ┌────────┐                       │
│ │ 主图   │  6933910412233   📋  │  69码（蓝色加粗14px，点击复制）
│ │ 120x120│  蜻蜓x1 PCIe4.0转M.2  │  产品名（白色12px，最多2行）
│ │ 占位图 │  🏷️ 转接卡            │  分类标签（自动从名称提取）
│ └────────┘                       │
└──────────────────────────────────┘
```

**交互**：
| 操作 | 行为 |
|------|------|
| 左键单击 | 选中（蓝色左边框），右侧加载该产品工作台 |
| 左键双击 | 打开该产品关联的文件夹 |
| 点击 69 码 / 📋 | 复制到剪贴板 + toast 提示 |
| 右键 | 菜单：复制69码 / 打开文件夹 |
| ↑↓ 键 | 键盘导航 |

**自动提取分类**：`_extract_category(name)` 根据产品名中是否含"转接卡/延长线/硬盘盒/..."等 20+ 关键词自动标注。分类用于卡片上的彩色标签。

**排序**：有关联产品的优先，同产品组的变体相邻排列。

**为什么以产品 SKU 为单位（不是产品组）**：
- 卖家的最小操作单元是 SKU（一个 69 码 = 一个可售卖的商品）
- 产品组（文件夹）是图片存储概念，不是业务概念
- 同一个文件夹下的颜色变体是不同 SKU，需要独立展示

#### 3.7.4 右侧：产品工作台（设计目标与架构）

**设计原则**：
1. **产品即工作台**：右侧不是被动图片浏览器，而是围绕一个产品的业务操作中心
2. **卡片式可扩展**：每个功能区块是独立的卡片（card），可折叠、可增删
3. **为知识库预留**：AI 功能卡片在知识库就绪后激活，不影响现有布局
4. **信息分层**：最常用的（图片、69码、规格）在上面，辅助功能在下面

**卡片架构（从上到下）**：

| 序号 | 卡片 | 功能 | 状态 |
|------|------|------|------|
| ① | **产品身份条** | 69码 + 名称 + 分类 + 操作按钮（固定，不滚动） | ✅ 已实现 |
| ② | **产品图片** | 按 image_set 分组展示缩略图网格，角标标注来源 | ✅ 已实现 |
| ③ | **规格参数** | 从 CSV/product_specs 读取的关键规格（可折叠） | ✅ 框架就绪 |
| ④ | **AI 工作台** | 基于知识库的智能功能（生成描述/翻译/卖点提取） | 🔮 阶段 7 |
| ⑤ | **关联信息** | 同组变体、所在文件夹、相似产品 | 🔮 阶段 7 |

**业务终极形态**（知识库就绪后）：
```
选中一个产品 → 右侧显示：
  ├─ 产品身份 + 快速复制
  ├─ 所有产品图片（按语言/用途分组）
  ├─ 规格参数（CSV + LLM 补充）
  ├─ 🤖 AI 工作台
  │   ├─ [生成英文描述] → 一键生成亚马逊 listing
  │   ├─ [提取卖点]     → 自动提取产品卖点
  │   ├─ [翻译为西语]   → 多语言市场支持
  │   ├─ [生成标题]     → 电商平台标题优化
  │   └─ [客服回复]     → 选中截图 + 产品信息 → AI 回复客户问题
  └─ 关联产品推荐
```

#### 3.7.5 交互流程

```
用户输入 "2233" (400ms 防抖)
  → SearchBar.search_triggered
    → SearchTab._on_search("2233")
      → db.search_products("2233")
        → 返回 [{"full_code":"6933910412233", "name":"蜻蜓x1...", ...}]
      → 为每个结果补充 category（_extract_category）和 main_image_path
      → product_list.set_products(results)
        → 渲染 ProductCard × N
        → 自动选中第一个 → 触发右侧加载

用户点击产品卡片
  → ProductCard.clicked(product_dict)
    → ProductList._on_card_clicked
      → 取消其他卡片选中态
      → 高亮当前卡片（蓝色左边框）
      → product_selected.emit(product)
        → SearchTab._on_product_selected(product)
          → gallery.load_images(images, nas_root)   — 加载图片网格
          → info_bar.set_product(product, nas_root)  — 更新身份条
          → specs_panel.set_product(product)         — 更新规格

用户点击 "📂 打开文件夹"
  → ProductInfoBar._on_open_folder
    → open_folder.emit(folder_path)
      → SearchTab._open_folder(path)
        → _safe_open_folder(path, nas_root)
          → _resolve_path → os.path.isdir → os.startfile

用户点击 69 码
  → 复制到剪贴板 → QToolTip 显示 "已复制 6933910412233"
```

#### 3.7.6 路径解析机制

**关键函数**（`search_tab.py` 顶部）：
```python
_resolve_path(relative, nas_root)   # 相对路径拼接 NAS 根目录
_safe_open_folder(path, nas_root)   # 安全打开文件夹（拼接→检查→startfile）
_safe_open_file(path, nas_root)     # 安全打开文件
```

**为什么需要这些函数**：
- 数据库存的是相对路径（如 `2.转接卡/蜻蜓系列/中文/01.jpg`）
- NAS 根目录在配置中（如 `K:\1.图片`）
- 打开文件夹/文件时必须拼接完整路径
- 早期 Bug：多处 `os.startfile` 直接用相对路径，导致"打开文件夹无效"

**路径拼接规则**：
- 已是绝对路径 → 直接返回
- 相对路径 + NAS 根目录已配 → `os.path.join(nas_root, relative)`
- 相对路径 + NAS 根目录未配 → 返回相对路径（调用方检查存在性）

### 3.8 扫描时扩展建设

扫描遍历是成本最高的一趟操作。以下数据在扫描时"顺便"提取：

| 扩展项 | 说明 | 存储 |
|--------|------|------|
| 文件夹名分词索引 | jieba 中文分词 + 空格切分英文 | `keyword_index` |
| 感知哈希 (pHash) | 64 位 pHash，用于去重 + "找相似图" | `images.phash` |
| 图片尺寸统计 | 宽、高、文件大小 | `images.width/height/file_size` |
| 错误日志 | 损坏/不支持的文件记录 | `data/scan_errors.log` |
| 文件夹变化追踪 | 每次扫描的增/删/改 | `scan_log` |

### 3.9 OCR 文字提取

- 使用 PaddleOCR 对每张图片提取文字
- 文字存入 `images.ocr_text`，支持后续关键字搜索
- 断点续扫：通过 `images.ocr_extracted` 标志位实现
- 增量扫描时仅对新图片执行 OCR
- 支持 80+ 语言（中/英/马来/泰/俄/韩/日 等）

### 3.10 AI 助手模块
- **形态**：独立标签页 `🤖 AI 助手`，与 `📦 图片检索` 并列。
- **功能**：
  - 全局快捷键触发截屏（全屏或区域），默认 `Ctrl+Shift+Z`，可配置。
  - PaddleOCR 提取屏幕文字。
  - 用户输入指令（如"帮我回复"、"总结要点"），连同 OCR 文字发送至 DeepSeek API。
  - 返回结果自动复制到剪贴板，并展示在助手面板中。
- **隐私**：截图仅在本地 OCR 处理，仅文本片段上传 API。
- **网络依赖**：AI 助手需要 DeepSeek API 连接，为核心检索之外的可选功能。

## 4. 数据设计

### 4.1 设计原则
- 产品组 `product_group` 是核心聚合单元
- 一个产品根文件夹 = 一个 `product_group`
- 一个 `product_group` 下可含多个 `products`（SKU 变体）
- 一个 `product_group` 下可含多个 `image_sets`（语言/用途子文件夹）
- 一个 `image_set` 下含多个 `images`
- 一个 `product` 可通过 `product_images` 关联多张图片
- JSONL 是产品库源文件，SQLite 是运行时编译产物

### 4.4 FAISS 向量索引
- 存储位置：`./data/faiss.index`
- 格式：`IndexHNSWFlat`，维度 = 512（CLIP ViT-B/32）
- 向量 ID 与 `images.id` 严格一致
- 仅对 `scan_priority = 1` 的图片做向量索引
- 索引元数据存储 `model_name`，加载时校验一致性

### 4.5 特征提取模型抽象层

```python
class BaseFeatureExtractor(ABC):
    @property
    def dim(self) -> int: ...
    @property
    def model_name(self) -> str: ...

    @abstractmethod
    def extract_batch(self, paths: list[str]) -> np.ndarray: ...
    @abstractmethod
    def extract_single(self, path: str) -> np.ndarray: ...
```

内置实现：
| 实现类 | 维度 | 模型名 | 说明 |
|--------|------|--------|------|
| `CLIPExtractor` | 512 | `"clip-ViT-B-32"` | 默认，CPU 平衡 |
| `CLIPLargeExtractor` | 768 | `"clip-ViT-L-14"` | 精度更高，慢 3x |

配置：`config.json` 字段 `feature_model: "clip-ViT-B-32"`。

### 4.6 模型切换逻辑

换模型 = 旧向量失效 → 提示重建：
  - `feature_extracted` 全部重置为 0
  - `faiss.index` 删除重建
  - 缩略图、pHash、keyword_index、产品数据均不受影响

## 5. 产品库与 LLM 前置识别

### 5.1 设计理念

将"AI 重活"和"扫描轻活"解耦：
  - LLM 图片分析：一次性、有网络、有成本
  - 扫描器：反复运行、离线、零成本
  - LLM 输出 `products.jsonl`，扫描器启动时导入 SQLite

### 5.2 products.jsonl 格式

每行一个 JSON 对象，代表一个产品组。

顶层字段：

| 字段            | 类型    | 必填 | 说明                          |
|-----------------|---------|------|-------------------------------|
| group_id        | string  | 是   | 产品组唯一标识                |
| folder_name     | string  | 是   | 产品根文件夹名                |
| folder_path     | string  |      | 相对路径                      |
| category        | string  | 是   | 一级分类                      |
| sub_category    | string  |      | 二级分类                      |
| status          | string  |      | active/discontinued/unsold    |
| common_specs    | object  |      | 组内所有变体共享的规格        |
| products        | array   | 是   | SKU 变体数组                  |
| image_sets      | array   |      | 图片集定义                    |
| shared_images   | array   |      | 多 SKU 共享图片               |
| data_source     | string  |      | llm/manual/csv                |
| llm_model       | string  |      | LLM 模型名                    |
| llm_timestamp   | string  |      | ISO 8601 时间                 |

products 数组元素：

| 字段            | 类型    | 说明                          |
|-----------------|---------|-------------------------------|
| product_id      | string  | SKU 唯一标识                  |
| name            | string  | 变体名称                      |
| full_code       | string  | 完整 69 码                    |
| code_tail       | string  | 后四位                        |
| sku_custom      | string  | 自定义 SKU                    |

image_sets 数组元素：

| 字段            | 类型    | 说明                          |

### 5.4 LLM 识别提示词关键要求

分析一个产品根文件夹时，LLM 必须：
  1. 识别文件夹下是否包含多个 SKU 变体
  2. 提取每个变体的区分规格（颜色、尺寸、版本等）
  3. 识别所有 image_set 子文件夹的类型和语言
  4. 判断哪些图片是多个变体共享的
  5. 输出严格 JSON，符合 products.jsonl 格式

### 5.5 CSV 与 JSONL 的匹配策略

当 LLM JSONL 中某个变体缺少 full_code 时：
  1. 用 matching_specs 在 CSV 中查找候选
  2. 匹配规则（按优先级）：
     - 名称中颜色一致 + 名称中规格一致
     - matching_specs 中所有字段一致
     - 部分字段一致（如仅颜色一致）
  3. 若唯一匹配 → 自动填充 full_code
  4. 若多个候选 → 按 score 排序，标记置信度 < 1.0
  5. 若无候选 → 标记 match_method = 'unmatched'

## 附录 C：产品规格动态表（v2.1 新增）

### C.1 设计原则

规格不应硬编码。不同类别的产品有不同的规格维度：

| 产品类别 | 关心的规格 |
|----------|-----------|
| 转接卡 | 接口版本(PCIe 4.0/5.0)、设备接口(M.2/NVMe/SATA)、支持尺寸(2280/2242) |
| 延长线 | 线长(25CM/50CM)、接口版本、方向(左进左出)、颜色 |
| 硬盘盒 | 硬盘尺寸(2.5寸/3.5寸)、盘位(单盘/双盘)、RAID、传输速度 |
| 散热片 | 颜色、材质、兼容尺寸 |
| 水晶头/护套 | 颜色、类别(六类/超五类)、包装数量 |

这些维度无法在编码时穷举。**应该在 CSV 导入和知识库导入时动态填充。**

### C.2 product_specs 表

```sql
CREATE TABLE product_specs (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    product_id  INTEGER NOT NULL,
    spec_group  TEXT    NOT NULL DEFAULT '',   -- 分组：物理参数/接口/报关信息/基本规格
    spec_name   TEXT    NOT NULL,              -- 规格名：长/宽/高/接口版本/颜色/...
    spec_value  TEXT    NOT NULL DEFAULT '',   -- 规格值
    source      TEXT    NOT NULL DEFAULT 'csv', -- csv / llm / auto / manual
    FOREIGN KEY (product_id) REFERENCES products(id)
);
```

### C.3 规格数据来源

| 来源 | 标记 | 触发时机 | 示例 |
|------|------|----------|------|
| CSV 导入 | `csv` | 导入产品 CSV 时，自动识别规格列 | 长/宽/高/重量/海关编码/开发员 |
| 名称自动提取 | `auto` | 搜索/展示时实时提取 | PCIe版本、颜色、线长 |
| LLM 知识库 | `llm` | 导入 products.jsonl 时 | 接口类型、支持尺寸、特性 |
| 手动编辑 | `manual` | 产品编辑面板（阶段 8） | 人工修正/补充 |

### C.4 CSV 列自动映射

导入 CSV 时，系统自动识别以下列名并映射到规格：

| CSV 列（模糊匹配） | spec_name | spec_group |
|-------------------|-----------|------------|
| 长(cm) / 长 / length | 长 | 物理参数 |
| 宽(cm) / 宽 / width | 宽 | 物理参数 |
| 高(cm) / 高 / height | 高 | 物理参数 |
| 商品净重 / 重量 / weight | 商品净重 | 物理参数 |
| 英文报关 / english_declare | 英文报关 | 报关信息 |
| 中文报关 / chinese_declare | 中文报关 | 报关信息 |
| 海关编码 / hs_code | 海关编码 | 报关信息 |
| 开发员 / developer | 开发员 | 其他 |

## 附录 D：独立知识库构建工具（v2.1 新增）

### D.1 设计理念

知识库构建与主应用**完全解耦**：

```
[build_knowledge_base.py]          ← 独立工具
  ├── 扫描 NAS 目录结构
  ├── 生成 LLM 提示词
  └── 输出 products.jsonl

[主应用 — 导入知识库]               ← 读取 JSONL 合并到 SQLite
  ├── 工具条 → "🧠 导入知识库"
  ├── 选择 .jsonl 文件
  └── 解析 → 更新 products + product_specs
```

**为什么解耦：**
- LLM 分析需要网络 + API Key + 费用，不适合嵌入离线优先的桌面工具
- JSONL 是纯文本，可版本控制、可手动编辑、可跨项目复用
- 不同 NAS 目录可用同一个知识库工具，只改输入路径

### D.2 使用流程

```bash
# 步骤 1：生成提示词
python build_knowledge_base.py --input K:\1.图片 --prompt-only

# 步骤 2：复制提示词发给 LLM（DeepSeek/GPT-4o），获得 JSONL 回复

# 步骤 3：保存为 products.jsonl

# 步骤 4：在主应用中导入
#   工具条 → 🧠 导入知识库 → 选择 products.jsonl
```

### D.3 LLM 提示词模板（核心）

提示词位于 `build_knowledge_base.py` → `build_prompt()` 函数。关键内容：

1. **识别产品根文件夹**：包含子文件夹的目录
2. **识别 SKU 变体**：从文件夹名提取 69 码后四位、型号、颜色、规格
3. **识别图片集**：子文件夹名 → set_type + language 映射表
4. **提取规格**：从文件夹名推断接口/尺寸/版本
5. **生成唯一 ID**：优先用 69 码，其次 SKU，最后 group_id + 序号

详细规范 → 见 `PRODUCT_JSONL_SCHEMA` 常量。

### D.4 JSONL 导入数据流

```
products.jsonl 每行 = 一个 product_group
  │
  ├── products[] → INSERT/UPDATE products 表
  │     ├── full_code → products.full_code (UNIQUE)
  │     ├── name → products.name
  │     ├── main_image_url → products.main_image_url
  │     └── code_tail = full_code[-4:]
  │
  ├── common_specs{} → INSERT product_specs (group级，所有变体共享)
  │     └── spec_name=接口, spec_value=PCIe 4.0 x1, source='llm'
  │
  └── products[].specs{} → INSERT product_specs (变体独有)
        └── spec_name=颜色, spec_value=黑色, source='llm'
```

### D.5 主图获取优先级

```
1. CSV 导入时下载的本地缓存 (products.main_image_local)
2. NAS 扫描的关联图片 (images.full_unc)
3. CSV 原始 URL (仅作标记，不下载时不使用)
```

| auto_specs | 从名称自动提取的规格数 | `_extract_specs_from_name()` |
| llm_specs | 知识库(JSONL)规格数 | products.jsonl |
| image_count | 关联的扫描图片数 | NAS扫描+product_folder_link |

完整度评分：● 三源完整 / ◑ 两个来源 / ◐ 单一来源 / ○ 无数据

### F.4 知识库生命周期

```
生成                  导入                  查询
build_knowledge_base  knowledge_tab         search_tab
  │                    │                     │

  ├→ AI返回JSONL        ├→ 更新product_specs

```
